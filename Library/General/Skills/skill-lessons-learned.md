SKILL: How to Prepare a Lessons Learned Report
VERSION: 1.0
DOMAIN: General
LEVEL: Base
JURISDICTION: Queensland

PURPOSE
A lessons learned report captures what worked, what did not, and what should be done differently on future projects or programs. It is a post-completion governance document that transfers knowledge from one project team to the organisation. A lessons learned report that sits in a project folder and is never read has no value. Its purpose is to improve future performance, not to document history.

WHEN TO USE THIS SKILL
- User asks to "prepare a lessons learned report" after a project or program
- User asks to "do a post-implementation review" or "post-project review"
- User needs to capture and share learnings from an event, incident, or program
- User asks to "review what went well and what didn't" after a significant piece of work
- User needs to meet a governance or audit requirement to complete a post-project review

STRUCTURE
Project overview: Name, period, total cost, key deliverables, and outcome against original objectives.
What worked well: Specific practices, decisions, or approaches that contributed to success. Tied to evidence, not opinion.
What did not work well: Specific problems, failures, or inefficiencies. Honest and factual. Not a blame exercise.
Root causes: For each significant issue, what caused it. Not symptoms — root causes.
Recommendations: Specific, actionable changes for future projects. Each recommendation should name who is responsible for implementing it and by when.
Risks that materialised: Which project risks actually occurred and what the impact was. Useful for future risk assessments on similar projects.
Knowledge to retain: Key contacts, relationships, documents, or technical knowledge that should be retained for future reference.

KEY PRINCIPLES
- Lessons learned reports must be honest. A report that only lists successes does not improve future performance.
- Root cause analysis is mandatory for significant issues. Describing a problem without identifying its cause produces a lesson that cannot be acted on.
- Recommendations must be specific and assigned. "Improve communication" is not a recommendation. "Implement a weekly steering committee update for all projects above $500,000, with a standard agenda template to be developed by the PMO by 30 June 2026" is a recommendation.
- The report should be completed while the team's memory is fresh -- ideally within four weeks of project completion.
- Share the report. A lessons learned report that is not distributed to relevant project managers and the PMO has not fulfilled its purpose.

QUALITY STANDARDS
A good lessons learned report produces at least three specific, actionable recommendations that are implemented on subsequent projects. If the recommendations are so general that they could apply to any project anywhere ("better planning", "clearer scope"), they will not change behaviour. The report should be proportionate to the project -- a $50,000 project does not need a 40-page PIR, but a $10M project does need a rigorous review.

QUEENSLAND CONTEXT
Lessons learned reports for Queensland local government projects should be retained as public records under the Public Records Act 2002. For capital works projects, the report should be provided to the asset owner and included in the asset file. Where an internal audit recommendation was made during the project, the lessons learned report should note whether the recommendation was implemented and what the outcome was. For projects above the significant procurement threshold, a post-contract evaluation may be required under the council's contracting framework. Lessons learned from projects involving privacy or information security incidents should be reported to the privacy officer and may require notification to the Office of the Information Commissioner.

EXAMPLE LANGUAGE
What did not work well: The scope of works for the Nerang depot upgrade was not agreed in writing before the contract was executed. This led to four scope variations totalling $380,000 (19% of original contract value) and a six-week program delay. The variations were individually approved within delegation but were not reviewed collectively until the third variation, at which point the cumulative cost and time impact had already crystallised.
Root cause: The project brief was approved at concept stage without a detailed scope document. The procurement team proceeded to market with a performance specification only, which gave the contractor significant latitude to interpret scope.
Recommendation: The Infrastructure Delivery team to develop a standard scope confirmation checklist for all construction projects above $200,000. The checklist to be signed by the project manager and contract manager before contract execution. PMO to implement by 31 August 2026.
