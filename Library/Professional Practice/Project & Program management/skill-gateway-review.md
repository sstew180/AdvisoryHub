SKILL: How to Conduct a Project Gateway Review
VERSION: 1.0
DOMAIN: Professional Practice
LEVEL: Base
JURISDICTION: Queensland

PURPOSE
A project gateway review is a structured assessment conducted at defined points in a project's lifecycle to determine whether the project is ready to proceed to the next phase. It is an independent check on whether the project has met the criteria required to proceed -- not a progress update and not a performance review. Gateway reviews are one of the most effective mechanisms for catching project problems early, before they become expensive.

WHEN TO USE THIS SKILL
- User asks to "conduct a gateway review" or "prepare for a gate review"
- User needs to assess whether a project is ready to proceed to the next phase
- User is preparing the documentation required for a gateway decision
- User is an independent reviewer preparing to assess a project at a gateway
- User asks to "review the project before we go to market" or "before construction starts"

STRUCTURE
Gate definition: Which gateway is being reviewed and what decision is required.
Review criteria: The specific criteria the project must meet to proceed. Defined before the review, not during it.
Evidence assessment: For each criterion, the evidence provided and whether it is sufficient.
Findings: What the review found -- against criteria, not in general.
Recommendation: Proceed / Proceed with conditions / Do not proceed.
Conditions (if applicable): Specific actions required before or after proceeding.
Review team: Who conducted the review and their independence from the project.

KEY PRINCIPLES
- Gateway reviews must be independent. A reviewer who is part of the project team cannot conduct an objective gateway review. Independence is what gives the review credibility.
- Criteria must be set before the review. Gateway criteria developed during the review to match the project's actual state are meaningless. Criteria come from the project framework or are agreed with the sponsor before the review begins.
- The recommendation must be one of three: proceed, proceed with conditions, or do not proceed. A gateway review that says "mostly okay" is not a gateway review.
- Conditions must be specific and trackable. "Improve the risk management plan" is not a condition. "Update the risk register to include the identified cybersecurity risks with treatment owners and due dates, and provide the updated register to the steering committee before contract execution" is a condition.
- Document the review. The review report is a governance record. It should be retained and referenced at subsequent gates.

QUALITY STANDARDS
A good gateway review is one that gives the decision-maker confidence that the project is ready for the next phase -- or clear evidence that it is not. If a project proceeds through a gateway and then fails in the next phase for a reason the gateway review should have caught, the review was inadequate. The independence and rigour of the review is what creates value.

QUEENSLAND CONTEXT
Gateway reviews for Queensland local government capital works projects should align with the council's project management framework and the Queensland Government's Project Assurance Framework where applicable. Projects above the significant procurement threshold are subject to the significant contracting plan requirements (LGR s221) and should include a gateway before going to market. For infrastructure projects funded by state or federal grants, the funding agreement may impose gateway review requirements. The QAO regularly reviews whether councils conduct appropriate project assurance on significant capital works -- the absence of gateway reviews is a common audit finding. Projects involving ICT should reference the Queensland Government's ICT project governance framework gateway criteria.

EXAMPLE LANGUAGE
Gate definition: This is a Gate 2 review (ready to proceed to market). The decision required is whether the project is sufficiently well-defined, scoped, and governed to proceed to the procurement phase.
Criterion 3: The project has an approved budget with identified funding source. Finding: The project charter references a budget of $4.2M. Budget allocation was confirmed in the 2025-26 council budget (line item Infrastructure Capital Works -- Fleet Management System, page 47). The budget includes a 15% contingency allowance. Sufficient evidence provided. Criterion met.
Criterion 5: Key risks have been identified, assessed, and have treatment owners. Finding: The project risk register contains 14 risks. However, three risks rated High have no treatment owner assigned and two have treatment due dates that have already passed. This criterion is not met. Condition required before proceeding to market: Risk register to be updated with treatment owners for all High risks and treatment due dates reviewed. Updated register to be provided to the steering committee chair for sign-off.
Recommendation: Proceed with conditions. The project may proceed to the procurement phase once the two conditions above are satisfied and confirmed by the steering committee chair.
